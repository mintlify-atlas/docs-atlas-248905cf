Llama Stack supports a broad set of remote inference providers beyond Ollama, OpenAI, vLLM, and Bedrock. All of them implement the same `Inference` API, so your application code stays the same regardless of which provider you use.

## Provider summary

| Provider type | Env vars required | Base URL / endpoint | Notes |
|---|---|---|---|
| `remote::anthropic` | `ANTHROPIC_API_KEY` | `https://api.anthropic.com` | Claude models |
| `remote::azure` | `AZURE_API_KEY`, `AZURE_API_BASE`, `AZURE_API_VERSION` | Your Azure resource URL | Azure OpenAI Service |
| `remote::cerebras` | `CEREBRAS_API_KEY` | `https://api.cerebras.ai/v1` | Cerebras Cloud |
| `remote::databricks` | `DATABRICKS_HOST`, `DATABRICKS_TOKEN` | Your workspace serving endpoint | Databricks model serving |
| `remote::fireworks` | `FIREWORKS_API_KEY` | `https://api.fireworks.ai/inference/v1` | Fireworks AI platform |
| `remote::gemini` | `GEMINI_API_KEY` | Google AI API | Google Gemini models |
| `remote::groq` | `GROQ_API_KEY` | `https://api.groq.com/openai/v1` | Ultra-fast Groq LPU |
| `remote::nvidia` | `NVIDIA_API_KEY` | `https://integrate.api.nvidia.com/v1` | NVIDIA NIM |
| `remote::runpod` | `RUNPOD_URL`, `RUNPOD_API_TOKEN` | Your RunPod endpoint URL | RunPod cloud GPU |
| `remote::sambanova` | `SAMBANOVA_API_KEY` | `https://api.sambanova.ai/v1` | SambaNova dataflow |
| `remote::together` | `TOGETHER_API_KEY` | `https://api.together.xyz/v1` | Together AI open-source models |
| `remote::vertexai` | `VERTEX_AI_PROJECT` | Google Cloud Vertex AI | Gemini via GCP; uses ADC |
| `remote::watsonx` | `WATSONX_API_KEY`, `WATSONX_PROJECT_ID` | `https://us-south.ml.cloud.ibm.com` | IBM WatsonX |

## Anthropic

Provides access to Claude claude-opus-4, Claude Sonnet, and Claude Haiku models.

```yaml
providers:
  inference:
    - provider_id: anthropic
      provider_type: remote::anthropic
      config:
        api_key: ${env.ANTHROPIC_API_KEY:=}
```

```bash
export ANTHROPIC_API_KEY=sk-ant-...
```

## Azure OpenAI

Azure OpenAI Service hosts GPT-4o and other OpenAI models in your Azure tenant.

```yaml
providers:
  inference:
    - provider_id: azure
      provider_type: remote::azure
      config:
        api_key: ${env.AZURE_API_KEY:=}
        base_url: ${env.AZURE_API_BASE:=}
        api_version: ${env.AZURE_API_VERSION:=}
        api_type: ${env.AZURE_API_TYPE:=}
```

```bash
export AZURE_API_KEY=...
export AZURE_API_BASE=https://your-resource-name.openai.azure.com/openai/v1
export AZURE_API_VERSION=2024-12-01-preview
export AZURE_API_TYPE=azure
```

<Note>
The `base_url` for Azure follows the pattern `https://<resource-name>.openai.azure.com/openai/v1`.
</Note>

## NVIDIA NIM

NVIDIA NIM provides optimized inference for NVIDIA-hosted models, including reranking models.

```yaml
providers:
  inference:
    - provider_id: nvidia
      provider_type: remote::nvidia
      config:

```

```bash
export NVIDIA_API_KEY=nvapi-...
```

NVIDIA also provides reranking endpoints. The provider maps rerank model identifiers to their specific API URLs automatically.

## Google Vertex AI

Vertex AI accesses Gemini models through Google Cloud and uses Application Default Credentials (ADC) instead of API keys.

```yaml
providers:
  inference:
    - provider_id: vertexai
      provider_type: remote::vertexai
      config:
        project: ${env.VERTEX_AI_PROJECT:=}
        location: ${env.VERTEX_AI_LOCATION:=global}
```

```bash
# Authenticate with GCP
gcloud auth application-default login

export VERTEX_AI_PROJECT=my-gcp-project
export VERTEX_AI_LOCATION=us-central1  # optional, defaults to global
```

Available models: `vertex_ai/gemini-2.0-flash`, `vertex_ai/gemini-2.5-flash`, `vertex_ai/gemini-2.5-pro`.

## Python example (any provider)

Once a provider is configured, the client code is identical regardless of which provider is active:

```python
from llama_stack_client import LlamaStackClient

client = LlamaStackClient(base_url="http://localhost:8321")

response = client.chat.completions.create(
    model="<model-id-registered-with-your-provider>",
    messages=[
        {"role": "user", "content": "Summarize the benefits of using a pluggable provider architecture."},
    ],
)
print(response.choices[0].message.content)
```
