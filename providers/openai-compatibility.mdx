Llama Stack implements the OpenAI API. You can use any OpenAI-compatible client — the official OpenAI Python or TypeScript client, LangChain, LlamaIndex, CrewAI, or any other framework — by pointing it at your Llama Stack server.

## Pointing an OpenAI client at Llama Stack

Llama Stack exposes OpenAI-compatible endpoints at `/v1`. For a server running locally on port `8321`, the full base URL is `http://localhost:8321/v1`.

<Tabs>
  <Tab title="OpenAI Python client">
    ```python
    from openai import OpenAI

    client = OpenAI(
        base_url="http://localhost:8321/v1",
        api_key="none",  # any non-empty string
    )

    response = client.chat.completions.create(
        model="meta-llama/Llama-3.2-3B-Instruct",
        messages=[{"role": "user", "content": "Hello"}],
    )
    print(response.choices[0].message.content)
    ```
  </Tab>
  <Tab title="Llama Stack client">
    ```python
    from llama_stack_client import LlamaStackClient

    # Set base_url to the root of your Llama Stack server (not /v1)
    client = LlamaStackClient(base_url="http://localhost:8321")

    response = client.chat.completions.create(
        model="meta-llama/Llama-3.2-3B-Instruct",
        messages=[{"role": "user", "content": "Hello"}],
    )
    print(response.choices[0].message.content)
    ```
  </Tab>
</Tabs>

<Note>
When using the Llama Stack client, set `base_url` to the server root (e.g., `http://localhost:8321`). The client routes OpenAI-compatible requests to the correct path automatically.

When using any other OpenAI client, set `base_url` to the `/v1` path (e.g., `http://localhost:8321/v1`).
</Note>

## Implemented endpoints

| API | Endpoint | Description |
|-----|----------|-------------|
| Chat completions | `POST /v1/chat/completions` | Text and vision inference, streaming, tool calling |
| Completions | `POST /v1/completions` | Text completions (legacy) |
| Embeddings | `POST /v1/embeddings` | Text embedding generation |
| Models | `GET /v1/models` | List available models |
| Files | `POST /v1/files`, `GET /v1/files` | File upload and management |
| Vector stores | `POST /v1/vector_stores`, `GET /v1/vector_stores` | Document storage and search |
| Batches | `POST /v1/batches`, `GET /v1/batches` | Offline batch processing |
| Responses | `POST /v1/responses` | Server-side agentic orchestration with tool calling, MCP, and file search |
| Conversations | `GET /v1/conversations` | Conversation state management |

## Responses API

The Responses API (`/v1/responses`) is the most feature-rich endpoint. It moves the agentic loop to the server, so a single API call can:

- Execute tool calls and feed results back to the model automatically
- Connect to any MCP server as a tool source
- Perform RAG via built-in file search against vector stores
- Maintain multi-turn conversation state with `previous_response_id`

```python
response = client.responses.create(
    model="meta-llama/Llama-3.2-3B-Instruct",
    input="Write a haiku about coding.",
)
print(response.output_text)
```

## Listing available models

```python
models = client.models.list()
for model in models:
    print(model.id)
```

## Endpoints not yet implemented

The following OpenAI API groups have no implementation in Llama Stack:

- `/assistants` — Assistants v2 API
- `/audio` — Speech, transcription, and translation
- `/fine_tuning` — Fine-tuning jobs
- `/images` — Image generation and editing
- `/realtime` — Realtime voice sessions
- `/threads` — Threads and runs (Assistants v1)
- `/organization` — Organization and project management

## Known limitations

<AccordionGroup>
  <Accordion title="Web search tool">
    The web search tool is partially implemented. Domain filtering, user location hints, and live-vs-cached control are not yet supported.
  </Accordion>
  <Accordion title="Reasoning content">
    The `reasoning.encrypted_content` include field exists but is a no-op. Preserving reasoning context across turns is not yet wired up to providers.
  </Accordion>
  <Accordion title="Response branching">
    Response branching (creating forks from a previous response ID) is not yet functional.
  </Accordion>
  <Accordion title="Service tier">
    The `service_tier` field on Responses is accepted but not passed through to inference providers.
  </Accordion>
  <Accordion title="Incomplete details">
    The `incomplete_details` field on response objects is not populated when generation stops early (e.g., at `max_output_tokens`).
  </Accordion>
  <Accordion title="Moderations category properties">
    The `/v1/moderations` endpoint returns a result, but the detailed per-category breakdown (harassment, hate, self-harm, sexual, violence, etc.) is not populated.
  </Accordion>
</AccordionGroup>

The conformance scores above are based on the Open Responses conformance test suite. For the latest scores, see the [Open Responses project](https://www.openresponses.org/).
