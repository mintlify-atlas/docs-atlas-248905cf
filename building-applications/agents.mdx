An agent in Llama Stack is a model-powered loop that can call tools, maintain conversation state across turns, and apply safety filters automatically. Use agents when a single LLM call is not enough.

<Tip>
  **Responses API vs Agents API** â€” For new applications, the **Responses API** (OpenAI-compatible) is the recommended approach. It supports dynamic configuration, conversation branching, and works with any OpenAI SDK. The legacy **Agents API** (sessions and turns) is still functional but not recommended for new projects. This page covers the `Agent` class, which is a Python client-side wrapper around the Responses API and is the easiest way to get started.
</Tip>

## Core concepts

### Agent configuration

Create an agent by passing a model, instructions, and tools:

```python
from llama_stack_client import Agent, LlamaStackClient

client = LlamaStackClient(base_url="http://localhost:8321")

agent = Agent(
    client,
    model="meta-llama/Llama-3-70b-chat",
    instructions="You are a helpful assistant that can use tools to answer questions.",
    tools=["builtin::code_interpreter", "builtin::file_search"],
)
```

| Parameter | Description |
|-----------|-------------|
| `model` | The underlying LLM |
| `instructions` | System prompt defining the agent's behavior |
| `tools` | Capabilities the agent can invoke |
| `input_shields` | Safety shields applied to user input |
| `output_shields` | Safety shields applied to agent responses |
| `max_infer_iters` | Maximum inference iterations per turn (default: 5) |

### Sessions

A session represents a conversation thread. Create one before sending messages:

```python
session_id = agent.create_session(session_name="my-conversation")
```

Sessions persist message history so the agent has context across turns.

### Turns

Each interaction is a "turn". A turn takes input messages and returns an output message along with every intermediate step the agent took.

<Tabs>
  <Tab title="Streaming">
    ```python
    from llama_stack_client import AgentEventLogger

    response = agent.create_turn(
        session_id=session_id,
        messages=[{"role": "user", "content": "Tell me about Llama models"}],
    )

    for log in AgentEventLogger().log(response):
        log.print()
    ```
  </Tab>
  <Tab title="Non-streaming">
    ```python
    from rich.pretty import pprint

    response = agent.create_turn(
        session_id=session_id,
        messages=[{"role": "user", "content": "Tell me about Llama models"}],
        stream=False,
    )

    print("Inputs:")
    pprint(response.input_messages)
    print("Output:")
    pprint(response.output_message.content)
    print("Steps:")
    pprint(response.steps)
    ```
  </Tab>
</Tabs>

## The agent execution loop

Each turn runs through a fixed sequence of steps:

<Steps>
  <Step title="Input safety check">
    If `input_shields` are configured, the user's message is screened before the model sees it.
  </Step>
  <Step title="Context retrieval">
    If RAG is enabled (via `file_search`), the agent queries the configured vector stores and injects retrieved context into the model's message history as a tool response.
  </Step>
  <Step title="Inference loop">
    The LLM generates a response. If the response contains tool calls:

    - Tool inputs pass through safety checks.
    - The tools execute (web search, code interpreter, file search, etc.).
    - Tool outputs are fed back to the LLM.
    - The loop repeats until the LLM returns a final response with no tool calls, the `max_infer_iters` limit is reached, or the token limit is exceeded.
  </Step>
  <Step title="Output safety check">
    If `output_shields` are configured, the agent's final response is screened before it is returned to the caller.
  </Step>
</Steps>

## Full example with tool calling

```python
from llama_stack_client import LlamaStackClient, Agent, AgentEventLogger

client = LlamaStackClient(base_url="http://localhost:8321")

agent = Agent(
    client,
    model="Llama3.2-3B-Instruct",
    instructions="You are a helpful assistant",
    tools=[
        {
            "name": "builtin::file_search",
            "args": {"vector_db_ids": ["my_docs"]},
        },
        "builtin::code_interpreter",
    ],
    input_shields=["llama_guard"],
    output_shields=["llama_guard"],
    max_infer_iters=5,
    sampling_params={
        "strategy": {"type": "top_p", "temperature": 0.7, "top_p": 0.95},
        "max_tokens": 2048,
    },
)

session_id = agent.create_session("monitored_session")

response = agent.create_turn(
    messages=[{"role": "user", "content": "Analyze this code and run it"}],
    documents=[
        {
            "content": "https://raw.githubusercontent.com/example/code.py",
            "mime_type": "text/plain",
        }
    ],
    session_id=session_id,
)

for log in AgentEventLogger().log(response):
    log.print()
```

## Responses API vs Agents API

Llama Stack has two APIs for agentic applications:

| | Responses API | Agents API |
|---|---|---|
| **SDK** | Standard OpenAI SDK | Llama Stack client only |
| **Configuration** | Dynamic per call | Static per session |
| **Conversation model** | Branching via response IDs | Linear sessions |
| **Tools** | `file_search`, `web_search`, MCP, functions | `builtin::file_search`, `code_interpreter` |
| **Safety** | Via `/v1/moderations` or guardrail params | Built-in `input_shields`/`output_shields` |

```python
from openai import OpenAI

client = OpenAI(base_url="http://localhost:8321/v1", api_key="fake")

# Dynamic configuration â€” change model or tools on every call
response = client.responses.create(
    model="llama3.2:3b",
    input="Search my docs for deployment instructions",
    tools=[{
        "type": "file_search",
        "vector_store_ids": ["vs_abc123"],
    }],
)

# Branch the conversation and switch models
response2 = client.responses.create(
    model="openai/gpt-4o",
    input="Now summarize what you found",
    previous_response_id=response.id,
)
```

For existing code that uses the Agents API (sessions and turns), it continues to work. For new projects, use the Responses API.

## Step types

Each turn in a session is made up of steps that you can inspect:

<AccordionGroup>
  <Accordion title="InferenceStep">
    The model generating text. Visible in non-streaming mode as an entry in `response.steps` with `step_type == "inference"`.
  </Accordion>
  <Accordion title="ToolExecutionStep">
    The agent calling a tool and receiving its output. Inspect `step.tool_calls` and `step.tool_responses` to see what was called and what came back.
  </Accordion>
  <Accordion title="ShieldCallStep">
    A safety shield evaluation. Contains the shield ID and whether a violation was detected.
  </Accordion>
</AccordionGroup>
